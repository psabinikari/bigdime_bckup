    Metadata API Design Analysis

    Skip to end of metadata
    • Created by Sabinikari, Pavan(AWF), last modified on Sep 17, 2015
    Go to start of metadata

    Problem Statement: 

    Metadata is the data that describes other data or in the other words it is all about "data about data". Source metadata schema elements are considered to be sent to metadata API in the JSON format by Adapters.  There is a need for current design to handle any kind of format or change in data order dynamically. Currently, metadata is associated independently inside each adapter and assumed to be static. The clear semantics of Adapter's data, as specified in its schema, should help metadata creators in easily creating effective metadata records in one central location. Below are the situations which provides the scope that we are trying to address.

    1. Change in the data order.
    2. Increase in the column count.
    3. Introducing new entity.
    4. Alter the data type.

    Depending on the future hassle points, we keep adding the situation that fit into this API. 

    Metadata API attempts to overcome the problem by storing the semantics of each meta data element of any schema.

    Analysis on the above problem statement:

    Design of metadata API more effectively and efficiently. This involves following data points with relevant observations, pros and cons.

    1. SDK(jar) vs REST API
    2. Data Storages.
    3. Locking Mechanisms.

    SDK(jar) vs REST API:

    Below are the two types of interfaces that would be proposed for the development of Metadata API.

     

    1. SDK: An API which is developed as a jar.
    2. REST API: An API which is developed as Rest web service.

     

    SDK_REST_API

     

    SDK:

    All the logic will be available as a jar. Hence it can be embedded into the required project (i.e., here in our case is Adapter projects) as a dependency jar file.

    Pros:

    1. Reduces the network traffic.
    2. Easy for error logging and debugging.
    3. Can be available as a dependency at the parent level for all the Applications.
    4. Once metadata api is embedded as jar, api capability is available to adapter. 

    Cons:

    1. Will have ensure that each adapter project contains latest or relevant version of code.
    2. Need to take care of common dependencies.

    REST API:

    All the logic will be located separately. Hence no need to think on dependency of other projects.

    Pros:

    1. Handled as an independent project.
    2. Deployed any where as long as network accessibility is provided.
    3. One version of code can be handled for all adapters.

    Cons:

    1. Increases network traffic.
    2. Hard to debug if there are any network issues.
    3. Need to ensure that Adapter can invoke the metadata api in the deployment environments.

    Analyisis on RDBMS vs NoSQL data bases:

    RDBMS Findings:

    1. Not good fit for large data volume(petabytes of data) with data types. Eg. images, Videos, text etc..
    2. Can't scale for large data volume.
    3. Scale-up - Limited by Memory processing(CPU) capabilities.
    4. Scale-out - Cache dependent 'Read' and 'Write' operations.
    5. Complex RDBMS model - Parsing, Locking, Logging, Buffer pool, Threads etc.
    6. Sharding causes operational problems e.g. managing a shard failure.
    7. consistency - A bottleneck for Scalability in RDBMS. Satisfying ACID is an hindrance for scaling. Relaxed consistency to scale out with NoSQL databases

    CAP Theorem

    CA: Single site cluster, therefore all nodes are always in contact. When a partition occurs, the system blocks.

    CP: Some data may not be accessible, but the rest is still consistent/accurate.

    AP: System is still available under partitioning, but some of the returned may be inaccurate.

    After observing all these findings and analysis, RDBMS should resolve our current purpose for storing Meta schema details for an Adapter/Source Application.

    Lock Mechanisms:

    Below are two types of locks available.

    1. Optimistic Lock.
    2. Pessimistic Lock.

    The disadvantage of pessimistic locking is that a resource is locked from the time it is first accessed in a transaction until the transaction is finished, making it inaccessible to other transactions during that time. If most transactions simply look at the resource and never change it, an exclusive lock may be overkill as it may cause lock contention, and optimistic locking may be a better approach.  With pessimistic locking, locks are applied in a fail-safe way.

     

    With optimistic locking, a resource is not actually locked when it is first is accessed by a transaction. Instead, the state of the resource at the time when it would have been locked with the pessimistic locking approach is saved. Other transactions are able to concurrently access to the resource and the possibility of conflicting changes is possible. At commit time, when the resource is about to be updated in persistent storage, the state of the resource is read from storage again and compared to the state that was saved when the resource was first accessed in the transaction. If the two states differ, a conflicting update was made, and the transaction will be rolled back.

    Hence better to choose optimistic lock.

    Hierarchical Lock Strategy:

    Adapters try to get/update the entire metadata JSON schema details from Metadata API and parse the incoming data. Hence there is no need for hierarchical lock for the current design.

    Explicit lock mechanism should be taken care if we decide the file system as our data store.

    If we decide to go with data base, optimistic locks are available at table, row or column level for most of the RDBMS or NoSQL data bases.

     

    Conclusion:

    Keeping all these observation, pros and cons. I have come with two designs.

    1. If dependency is not a constraint, I would suggest to have Adapter with SDK and data storage RDBMS(MySQL).
    2. If dependency is a constraint, I would suggest to have Adapter with REST API and data storage as File System.

    Since we already have MYSQL in our Big dime framework we would like to go with (1) where we don't need to take care of explicit file location mounting and file lock mechanism.  

    Please feel free to provide your inputs and comments.
